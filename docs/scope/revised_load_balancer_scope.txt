==============================================================
LOAD BALANCER (simple) -- REVISED PROJECT SCOPE
Capstone: Systems Programming in Python
Stack: Python CLI / daemon / SQLite DB
Optional Web Layer: Django + HTMX only if a dashboard is added
==============================================================

GOAL
----
Build a fully functional learning load balancer that runs as a long-lived
Python daemon, accepts client TCP connections, selects healthy backends using
pluggable balancing algorithms, relays bytes bidirectionally, records metrics,
exposes a CLI/admin control plane, supports graceful drain/reload, and includes
intentional capstone features for parsing, process control, graph-style
algorithms, bytecode-style execution, and cryptographic primitives.

This is a serious student capstone, not a production load balancer. It should
be ambitious, coherent, testable, and honest about its limits.

Concepts genuinely in play:
  - SYSTEMS THINKING: sockets, byte streams, hot path design, liveness,
    concurrency, backpressure, timeouts, failure modes
  - PROCESS CONTROL: long-running daemon, PID file, signals where supported,
    graceful shutdown, graceful drain, config reload, foreground service mode
  - PARSING: typed config file parser, CLI command parsing, admin protocol
    framing, optional routing-rule DSL
  - GRAPH / HASH-RING ALGORITHMS: consistent hashing with virtual nodes for
    sticky backend selection
  - CRYPTOGRAPHIC PRIMITIVES: HMAC-signed admin commands, secure tokens, and
    optional TLS wrapping with Python's ssl module
  - BYTECODE / VM THINKING: compile a tiny rule DSL into internal instructions
    and interpret those instructions for admin/routing/policy decisions

Legend:
  [CORE] = required for the capstone
  [STRETCH] = optional polish after the core works end-to-end

ACCEPTANCE CRITERIA
--------------------------------------------------------------
[CORE]    Balancer accepts TCP client connections and relays bytes to local
          echo backends end-to-end.
[CORE]    Round-robin distributes new connections across healthy eligible
          backends in stable order.
[CORE]    Weighted round-robin honors backend weights over a large sample.
[CORE]    Least-connections selects the healthy backend with the fewest active
          connections under concurrent load.
[CORE]    Consistent hashing keeps most sticky clients mapped to the same
          backend after a backend is added or removed.
[CORE]    Active health checks mark a backend unhealthy after N failures and
          healthy again after M successes.
[CORE]    Passive failures are recorded and influence health suspicion without
          instantly flapping the pool.
[CORE]    A draining backend receives no new traffic while existing connections
          continue until completion or drain timeout.
[CORE]    Reload applies a valid new config without dropping existing
          connections; invalid reloads are rejected and the previous good
          config remains active.
[CORE]    HMAC-signed admin commands succeed; tampered, expired, and replayed
          commands fail cleanly.
[CORE]    The admin protocol parser rejects malformed, oversized, unsigned, or
          unsupported-version frames.
[CORE]    Rule DSL source parses, compiles to internal bytecode, executes
          deterministically, and refuses unsafe/invalid rules.
[CORE]    SQLite config/metrics/history work, but no SQLite query or write blocks
          the per-connection hot path.
[CORE]    Graceful shutdown stops accepting new connections, drains active
          connections up to timeout, then exits cleanly.
[CORE]    Failure cases such as no healthy backend, backend connect failure,
          idle timeout, bad config, bad HMAC, and DB write pressure do not crash
          the daemon.


DESIGN CALLS
------------
[CORE]    Layer 4 TCP proxying is the main product. The balancer accepts a
          client connection, chooses a backend, connects to it, and relays raw
          bytes in both directions. This keeps the hot path protocol-agnostic.

[CORE]    Layer 7 HTTP routing is optional. If added, it must be a separate
          mode layered on top of the L4 balancer, not a rewrite of the core.

[CORE]    The DB is for configuration snapshots, metrics, events, and history.
          It must never be queried on the per-connection hot path. Config is
          loaded at startup/reload into memory; metrics are written
          asynchronously or on a controlled interval.

[CORE]    Keep selection logic pure and testable. Given a backend pool and
          runtime state, a strategy returns a backend. Sockets, DB writes, CLI
          parsing, and daemon concerns wrap around this clean core.

[CORE]    Use Python's standard library heavily where reasonable: socket,
          asyncio, signal, argparse, sqlite3, logging, hmac, hashlib,
          secrets, ssl, dataclasses, typing, enum, time.monotonic.

[CORE]    Final concurrency decision: implement the balancer with asyncio
          streams/tasks as the primary concurrency model. The README should
          compare asyncio with selectors and thread-per-connection, but the
          implementation should be one coherent async design, not a mix of
          unrelated models.

1A. CORE CONNECTION LIFECYCLE & RELAY SEMANTICS
--------------------------------------------------------------
[CORE]    Model each proxied connection with an explicit lifecycle:
            ACCEPTED
            SELECTING_BACKEND
            CONNECTING_BACKEND
            RELAYING
            HALF_CLOSED_CLIENT
            HALF_CLOSED_BACKEND
            CLOSING
            CLOSED
            FAILED

[CORE]    Each connection record tracks:
            - connection_id
            - client address
            - selected backend id
            - accepted monotonic timestamp
            - last activity monotonic timestamp
            - bytes client_to_backend
            - bytes backend_to_client
            - close reason
            - error summary if applicable

[CORE]    Active backend connection accounting:
            - increment only after backend connection succeeds
            - decrement exactly once during final cleanup
            - cleanup must be idempotent
            - counters must remain correct when tasks are cancelled, time out,
              or fail during relay

[CORE]    Relay implementation:
            - create one task for client -> backend and one task for backend -> client
            - update byte counters on successful reads/writes
            - update last-activity timestamp on successful traffic
            - enforce idle timeout from monotonic time
            - if one side reaches EOF, half-close/write_eof to the other side
              where supported by asyncio transports
            - cancel sibling relay task only when teardown requires it
            - always close both sides during final cleanup
            - record close reason: client_closed, backend_closed, timeout,
              connect_failed, shutdown, no_backend_available, or error

[CORE]    NoBackendAvailable behavior:
            - close the client connection cleanly
            - record a failed selection event
            - increment a global no_backend_available counter
            - do not crash the daemon
            - make the behavior observable in logs, metrics, and status output


--------------------------------------------------------------
2. BALANCING STRATEGIES
--------------------------------------------------------------
[CORE]    Pluggable strategy interface:
            - name
            - select(pool, connection_context) -> backend
            - optional on_success/on_failure hooks for passive feedback

[CORE]    Round-robin:
            - cycles through healthy eligible backends in stable order
            - skips unhealthy/draining/disabled backends
            - does not get stuck when the pool changes during reload

[CORE]    Weighted round-robin:
            - higher-weight backends receive proportionally more selections
            - distribution must be tested over a large sample

[CORE]    Least-connections:
            - selects the healthy backend with the fewest active connections
            - active counts increment only after backend connection success
            - counts decrement exactly once on teardown
            - concurrency-safe under simultaneous connects/closes

[CORE]    Consistent hashing:
            - implement a hash ring with virtual nodes
            - route sticky clients by source IP in L4 mode
            - in optional L7 mode, allow sticky routing by header/cookie/path key
            - demonstrate that adding/removing a backend minimizes remapping
            - explain why this is a graph/ring-style distribution algorithm

[CORE]    Exact strategy behavior:
            - If no eligible backend exists, raise NoHealthyBackendError and
              trigger the NoBackendAvailable behavior.
            - Round-robin keeps an index into the stable eligible backend list
              and advances once per successful selection attempt.
            - Weighted round-robin uses smooth weighted round-robin:
                  current_weight += weight for each eligible backend
                  select backend with highest current_weight
                  selected.current_weight -= total_eligible_weight
            - Least-connections tie-breaker: choose the lowest active count;
              if tied, use round-robin among tied candidates.
            - Consistent hashing ring points are generated from
              sha256(backend_id || virtual_node_index); each backend has
              weight * virtual_nodes_per_weight ring points; lookup chooses
              the first point >= hash(sticky_key), wrapping to the beginning.
            - Consistent hashing tests must measure remapping percentage when
              adding/removing a backend.

[STRETCH] Random and weighted-random strategies.

[STRETCH] Strategy plugin loading by import path.


--------------------------------------------------------------
3. TCP PROXY HOT PATH
--------------------------------------------------------------
[CORE]    Accept client connections on a configured listener host/port.

[CORE]    Build a connection context:
            - client address
            - accepted timestamp from time.monotonic()
            - requested strategy
            - optional sticky key

[CORE]    Select a backend through the configured strategy.

[CORE]    Connect to the selected backend with a timeout.

[CORE]    Relay bytes bidirectionally:
            - client -> backend
            - backend -> client
            - continue until either side closes, errors, or times out
            - handle half-close behavior as cleanly as the chosen socket model
              allows

[CORE]    Enforce timeouts:
            - backend connect timeout
            - idle timeout
            - graceful shutdown drain timeout
            - health-check timeout

[CORE]    Backend-connect failure handling:
            - if the chosen backend fails, try the next eligible backend
            - avoid infinite retry loops
            - record failure event
            - feed passive health-check suspicion

[CORE]    Correct accounting:
            - active connections
            - total handled
            - bytes proxied in both directions
            - failure counts
            - per-backend and global counters

[CORE]    Hot path must not:
            - query SQLite
            - block on metrics flush
            - parse config files
            - call the admin CLI


--------------------------------------------------------------
4. HEALTH CHECKS AND LIVENESS
--------------------------------------------------------------
[CORE]    Active health checks:
            - background task/thread separate from the hot path
            - monotonic scheduling
            - TCP connect probe by default
            - optional HTTP GET /health in L7 mode

[CORE]    Hysteresis:
            - backend becomes unhealthy after N consecutive failures
            - backend returns healthy after M consecutive successes
            - prevents single blips from flapping the backend in/out

[CORE]    Passive health checks:
            - live connection failures mark a backend suspect
            - passive failures can accelerate active re-checks
            - passive checks should not instantly remove a backend unless the
              configured threshold is met

[CORE]    Health state transitions:
            - log all transitions
            - write transition events to SQLite
            - expose in status output

[CORE]    Draining:
            - a draining backend receives no new connections
            - existing proxied connections are allowed to finish
            - when active count reaches zero, removal can complete safely


--------------------------------------------------------------
5. DAEMON AND PROCESS CONTROL
--------------------------------------------------------------
[CORE]    Run modes:
            - foreground mode for development and Windows compatibility
            - daemon/service mode where supported

[CORE]    PID file:
            - write PID on start
            - refuse duplicate daemon starts when PID is alive
            - clean up PID file on normal exit

[CORE]    Shutdown:
            - SIGINT/SIGTERM on Unix-like systems
            - CLI stop command through control socket on all platforms
            - stop accepting new connections
            - mark balancer as draining
            - wait for in-flight connections up to a configured timeout
            - force close remaining connections after timeout

[CORE]    Reload:
            - SIGHUP on Unix-like systems where available
            - CLI reload command through control socket on all platforms
            - reload config from file/DB snapshot
            - apply new backend list, weights, strategy, and health settings
            - do not drop existing connections
            - removed backends enter draining state first

[CORE]    Control socket:
            - local TCP or Unix-domain socket where supported
            - framed command protocol
            - commands are authenticated with HMAC
            - used by CLI for stop, reload, status, drain, enable, disable,
              add/remove backend, and strategy changes

[CORE]    Admin protocol concrete frame format:
            - length-prefixed JSON frames
            - first 4 bytes: unsigned big-endian frame length
            - followed by UTF-8 JSON payload bytes
            - reject frames over a configured max size

[CORE]    Admin payload shape:
            {
              "version": 1,
              "timestamp": 1710000000,
              "nonce": "base64-or-token",
              "command": "status",
              "args": {},
              "signature": "hex-or-base64-hmac"
            }

[CORE]    HMAC input uses canonical JSON:
            - UTF-8
            - sorted keys
            - compact separators
            - signature field excluded from signed bytes
            - include version, timestamp, nonce, command, and args

[CORE]    Admin parser rejects:
            - oversized frames
            - malformed JSON
            - missing required fields
            - unsupported version
            - expired timestamp
            - replayed nonce
            - invalid signature
            - unknown command
            - invalid argument types

CAPSTONE PRIORITIES
--------------------------------------------------------------
The project should remain ambitious. Do not reduce the core to only a toy
round-robin proxy. The required capstone signal is:

  1. A real L4 TCP proxy with bidirectional byte relay.
  2. Correct least-connections under concurrency.
  3. Weighted round-robin and consistent hashing.
  4. Active and passive health checks with hysteresis.
  5. Graceful drain, graceful shutdown, and live reload.
  6. SQLite-backed config snapshots and metrics history, off the hot path.
  7. HMAC-authenticated control commands.
  8. A parsed config file and framed admin protocol.
  9. A tiny DSL compiled to internal bytecode and interpreted safely.
 10. A credible CLI and test suite.

Build the full capstone in layers, but keep the final target complete.

REVISION NOTE:
This revised scope keeps the original core ambitious and intact. The additions
make implementation behavior more concrete: acceptance criteria, asyncio as the
